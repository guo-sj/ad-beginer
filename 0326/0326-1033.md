  ---
  AdComponent 拉起 UEA 的完整逻辑

  AdComponent 里存在两种 Ability，要区分清楚：

  ┌──────────────────────────┬────────────────────────┬──────────────────────────────────┐
  │         Ability          │        配置 key        │               用途               │
  ├──────────────────────────┼────────────────────────┼──────────────────────────────────┤
  │ ServiceExtensionAbility  │ providerApiAbilityName │ RPC 通信（点击/曝光/自渲染事件） │
  ├──────────────────────────┼────────────────────────┼──────────────────────────────────┤
  │ UIExtensionAbility (UEA) │ providerUEAAbilityName │ 渲染广告 UI                      │
  └──────────────────────────┴────────────────────────┴──────────────────────────────────┘

  UEA 不是通过 connectServiceExtensionAbility 拉起的，而是由 UIExtensionComponent
  渲染时隐式拉起的（UIExtensionComponent.create(this.want)，want.abilityName 指向 providerUEAAbilityName）。

  ---
  触发 UEA 的调用链

  aboutToAppear()
    └─ getConfigJsonData()         // 读配置文件，构建 map
         └─ setWant()              // 设置 this.want（含 UEA abilityName）
              └─ initAdRender()   // 判断走哪条路

  initAdRender 是核心分叉点（第 348 行）：

  路径一：普通广告（非自渲染）

  initAdRender() — else 分支
    └─ this.showComponent = true   ← 立即触发渲染
    └─ connectServiceExtAbility()  ← 连接 ServiceExt（异步，可能晚于 UEA）
  showComponent 变为 true → initialRender 里的 If 条件成立 → updateView() → adRenderer.bind(this)() → Component() →
  UIExtensionComponent.create(want) → UEA 被拉起

  路径二：自渲染广告（adType===3 && canSelfRendering && creativeType!==99）

  initAdRender() — if 分支
    └─ this.isAdRenderer = true
    └─ connectServiceExtAbility()  ← 先连 ServiceExt
         └─ onConnect 回调
              └─ this.showComponent = true  ← 等 ServiceExt 连上才触发渲染
  此时如果 adRenderer 是默认的 this.Component，同样会拉起 UEA；
  如果外部传入了自定义 adRenderer（真正自渲染），则不使用 UIExtensionComponent，UEA 不被拉起。

  ---
  一句话总结

  ▎ UEA 在 showComponent 变为 true 时被拉起。普通广告在 initAdRender 里立即设置；自渲染广告则等 ServiceExt onConnect 回调后才设置。控制
  showComponent 的地方就是控制 UEA 拉起时机的地方。

  ---
  原生广告请求为什么能拉起 UEA

  先澄清一个关键区别：框架里拉起 UEA 的入口有两个，它们针对不同的广告类型。

  ---
  入口一：showAd — Modal 方式（C++ 层）

  这是针对开屏、插屏、激励广告等全屏广告的路径，调用链：

  JS: advertising.showAd(ad, displayOptions, context)
    └─ C++: Advertising::ShowAd()         // advertising.cpp:525
         └─ ShowAdHandler()               // advertising.cpp:500
              └─ want.SetElementName(bundleName, ueaAbilityName)   // 用 providerUEAAbilityName
              └─ StartUIExtensionAbility()  // advertising.cpp:440
                   └─ uiContent->CreateModalUIExtension(want, ...)  // 以 Modal 窗口拉起 UEA

  关键点：want 里的 abilityName 来自配置文件的 providerUEAAbilityName，通过 CreateModalUIExtension 以模态窗口方式拉起 UEA，会覆盖在当前 UI 上方。

  ---
  入口二：AdComponent — 嵌入式方式（JS 层）

  这是针对信息流（原生）广告的路径，上一个问题已分析过：

  showComponent = true
    └─ updateView() → adRenderer() → Component()
         └─ UIExtensionComponent.create(want)   // want.abilityName = providerUEAAbilityName
              └─ 框架自动连接 UEA（嵌入式，非模态）

  ---
  loadAd 请求本身不拉起 UEA

  AdLoader.loadAd()
    └─ C++: AdLoadService::LoadAd()
         └─ ConnectAdKit()
              └─ ConnectAbility(extensionName)   // 连的是 ServiceExtension（providerAbilityName）
                                                 // 不是 UEA！

  loadAd 阶段只连 ServiceExtensionAbility，负责广告数据的请求与回调，和 UEA 完全无关。

  ---
  总结对比

  ┌─────────────────────────┬────────────────────────────┬────────────┬───────────────────────────────────┐
  │          场景           │         谁拉起 UEA         │    方式    │             配置字段              │
  ├─────────────────────────┼────────────────────────────┼────────────┼───────────────────────────────────┤
  │ showAd (开屏/插屏/激励) │ C++ CreateModalUIExtension │ Modal 窗口 │ providerUEAAbilityName            │
  ├─────────────────────────┼────────────────────────────┼────────────┼───────────────────────────────────┤
  │ AdComponent (信息流)    │ ArkUI UIExtensionComponent │ 嵌入式     │ providerUEAAbilityName            │
  ├─────────────────────────┼────────────────────────────┼────────────┼───────────────────────────────────┤
  │ loadAd 请求阶段         │ 不拉 UEA                   │ -          │ providerAbilityName（ServiceExt） │
  └─────────────────────────┴────────────────────────────┴────────────┴───────────────────────────────────┘
