  ---
  1. 多个 AdComponent 只能监听到一个组件的点击事件

  原因：第二个组件的渲染器永远不会显示。

  流程如下：

  initAdRender() 中，isAdRenderer = true 时，只设置 Behavior/initIds，不设置 showComponent = true，而是靠 sendDataRequest(INIT) 的 reply 来触发：

  // sendDataRequest 内部
  if (u === INIT) {
      this.showComponent = true;  // 只有 INIT 成功才显示渲染器
  }

  但第二个组件调用 connectServiceExtAbility() 时，命中这个守卫直接 return：

  // adcomponent.js:344
  if (AdConnectionManager.connection !== -1 || AdConnectionManager.connectCount > 0) {
      return;  // 第二个组件直接跳出
  }

  onConnect 永远不会为第二个组件触发，sendDataRequest(INIT) 永远不被调用，showComponent 保持 false，initialRender() 里的 if (this.showComponent)
  分支不走，Column 和它上面的 onClick 根本没有被渲染出来。

  结论：第二个组件不是"点击没响应"，而是"根本没渲染出来可以点击的区域"。

  ---
  2. connectCount 的冲突问题

  ArkTS 是单 JS 线程，没有真正的多线程竞争，但 async/await 的挂起会导致逻辑上的交错。这里有一个更根本的设计问题：

  connectCount 根本没有正确计数。

  它的设计意图明显是引用计数：多个组件共用一个连接，全部销毁时才真正 disconnect。但实现有漏洞：

  // 第二个组件进来，直接 return，没有 connectCount++
  connectServiceExtAbility() {
      if (AdConnectionManager.connection !== -1 || ...) {
          return;  // ← 第二个组件在这里退出，connectCount 仍是 1
      }
  }

  // connectCount 只在 onConnect 里 +1，只有第一个组件的 onConnect 会触发
  AdConnectionManager.connectCount++;  // 最终值永远是 1

  结果：

  ┌───────────────────────────┬──────────────────┬──────────────────┐
  │           时序            │   connectCount   │       状态       │
  ├───────────────────────────┼──────────────────┼──────────────────┤
  │ 组件A连接成功             │ 1                │ 正常             │
  ├───────────────────────────┼──────────────────┼──────────────────┤
  │ 组件B跳过连接             │ 1（本该是2）     │ 少计             │
  ├───────────────────────────┼──────────────────┼──────────────────┤
  │ 组件A销毁，disconnect     │ 0 → reset()      │ remoteObj 被清空 │
  ├───────────────────────────┼──────────────────┼──────────────────┤
  │ 组件B尝试 sendDataRequest │ remoteObj = null │ 崩溃             │
  └───────────────────────────┴──────────────────┴──────────────────┘

  另一个 async 交错问题：disconnectServiceExtAbility() 里先同步减 connectCount，然后才 async 调用系统 disconnect。在 finally 之前、async
  挂起期间，如果另一个 disconnect 也进来执行，connectCount 会被多次减，可能导致提前 reset()。

  正确做法：第二个组件走"复用连接"路径时，也要 connectCount++；销毁时对称地 connectCount--，只有减到 0 才真正 disconnect。
