  ---
  问题一：为什么实际运行时触发 3 次 onRemoteRequest, code: 3

  根本原因：CUSTOMIZED_RENDERING = 3 是所有业务事件上报共用的同一个 RPC 方法码。

  看 sendDataRequest 的实现（autoadcomponent.js:218）：

  await this.remoteObj.sendMessageRequest(CUSTOMIZED_RENDERING, data, reply, option)
  //                                       ↑ 固定是 3

  无论实际事件类型是什么（SHOW_START=0、IMP=1、CLICK=2），全部通过 sendMessageRequest(code=3, ...) 发送，真正的事件类型通过 data.writeInt(u) 写进 payload 里。Provider 侧的 onRemoteRequest
  只能看到方法码 3，无法直接区分事件类型。

  触发 3 次的时序（以 canSelfRender=true 的 AutoAdComponent 为例）：

  ┌──────┬──────────────────────────────────────────────────────────┬────────────────────┬────────────┐
  │ 次序 │                         触发位置                         │ 事件类型 (payload) │ 服务端看到 │
  ├──────┼──────────────────────────────────────────────────────────┼────────────────────┼────────────┤
  │ #1   │ onConnect → sendDataRequest(SHOW_START)                  │ 0                  │ code: 3    │
  ├──────┼──────────────────────────────────────────────────────────┼────────────────────┼────────────┤
  │ #2   │ visibleAreaChange(true, o≥0.9) → sendDataRequest(IMP, o) │ 1                  │ code: 3    │
  ├──────┼──────────────────────────────────────────────────────────┼────────────────────┼────────────┤
  │ #3   │ onAreaClick() → sendDataRequest(CLICK)                   │ 2                  │ code: 3    │
  └──────┴──────────────────────────────────────────────────────────┴────────────────────┴────────────┘

  三种不同的业务事件（展示开始、曝光、点击）全部复用同一个 RPC 方法码 3，所以 Provider 侧连续收到 3 次 onRemoteRequest, code: 3。

  问题二：为什么 aboutToDisappear 之前乃至整个过程都没有触发 handleUIFail

  因为 handleUIFail 是异常兜底机制，不是正常生命周期回调。

  整个机制的设计如下：

  客户端 (AutoAdComponent)                     Provider 服务端
      │                                              │
      ├─ registerFailCallback()                      │
      │   sendMessageRequest(SET_CALLBACK=6, stub)──►│  服务端保存 failCallbackStub
      │                                              │
      │  正常运行阶段（服务一直存活）                    │
      │  ← 服务端不会主动回调 stub ──────────────────  │
      │                                              │
      ├─ aboutToDisappear()                          │
      │   ① unregisterFailCallback()                 │
      │      sendMessageRequest(SET_CALLBACK=6,0)──►│  服务端删除 failCallbackStub
      │   ② disconnectServiceExtAbility()──────────►│  正常断开

  handleUIFail 的触发路径是：

  服务端检测到 UIExtension 进程崩溃
      → 服务端主动调用 failCallbackStub
      → 客户端 onRemoteMessageRequest() 被回调
      → this.component.handleUIFail()

  整个正常测试过程中不触发的原因：

  1. aboutToDisappear 之前：UIExtension 服务运行正常，未崩溃，服务端从不主动调用 failCallbackStub，因此客户端的 onRemoteMessageRequest 永远不被唤醒。
  2. aboutToDisappear 期间：代码先执行 unregisterFailCallback()（移除了 stub），再执行 disconnectServiceExtAbility()，即便断开过程产生任何异常，failCallbackStub
  已经注销，服务端也无法再回调它。
  3. 根本设计意图：handleUIFail 只处理 UIExtension 进程意外死亡这一种故障场景（日志信息 'UI extension died.' 也印证了这一点）。正常用户关闭页面走的是 aboutToDisappear → unregister →
  disconnect 的优雅退出路径，不需要也不应该触发 handleUIFail。
